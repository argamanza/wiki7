# Phase 3b Backlog

Collection point for content / template / structure / data-quality gaps observed during the Phase 3a prod push (and beyond). Phase 3a's scope is single-season pipeline-to-prod end-to-end; everything richer than "renders without errors and the data is there" lands here.

When 3a closes, this file becomes the prioritised work list for Phase 3b.

## How to file an entry

One bullet per gap. Don't pre-judge severity — that comes during 3b planning.

```
- **<terse title>** *(category)* — what's wrong + where it was seen. Optional: `// hypothesis: <suspected cause>` or `// fix: <one-liner>`.
```

Categories (pick the closest; cross-cutting items go under `Cross`):

| Category | Covers |
|---|---|
| `Templates` | MediaWiki `Template:` pages (infoboxes, navboxes) and the Jinja `.j2` templates in `data/wiki_import/templates/` that render to wikitext. |
| `Cargo` | `#cargo_declare` schema design, `#cargo_store` calls inside templates, missing queries on aggregate pages. |
| `Structure` | Page sections missing / in wrong order / too thin; homepage TODO placeholders; navigation / cross-links between page types. |
| `Data` | Pipeline data gaps (missing fields, wrong values, untranslated strings — *not* the auto-translate review of `mappings.he.yaml` itself, which is its own recurring task). |
| `SEO` | Per-page `{{#seo:}}` overrides, alt tags, URL slug strategy, internal-linking quality once Cargo queries are populated. |
| `Cross` | Items that span buckets (e.g. "player infoboxes need both a Cargo declare AND a `{{#seo:}}` block AND an alt-tag wiring"). |

Spot-check protocol: when you see something, just paste the URL or page title + a one-line description. I'll formalise it into the table on commit.

- **`getUserPermissionsErrors` hook enforcing same-namespace-only moves for Wiki7Bot** *(Docker — Phase 3b; spec **corrected 2026-06-13** during the Helder-Lopes mapping-fix trace)* — the §6 ⑤ fix granted Wiki7Bot the `move` right globally. Pattern A's invariant is "never CROSS namespaces" (never auto-promote Draft→main, never auto-demote main→Draft — the catastrophic case that hides a live page); same-namespace title-drift renames are LEGITIMATE and occur in BOTH namespaces (fixing a malformed Hebrew name renames `Draft:X → Draft:Y`, OR `X → Y` in mainspace for a promoted page). The hook must mirror that exactly: deny a bot move only when `$oldTitle->getNamespace() !== $newTitle->getNamespace()` (block cross-namespace promote/demote), and ALLOW same-namespace renames. **Do NOT restrict to NS_DRAFT-only** — that would block the legitimate mainspace rename that happens when a reviewer fixes a promoted page's name via `mappings.he.yaml` (real, recurring case caught during the Helder-Lopes trace). Rationale: a cross-namespace move can hide a live page (irrecoverable); a same-namespace rename is visible, move-logged, and reversible. Effort: ~30 lines + a test asserting BOTH directions (same-ns allowed, cross-ns denied).

- ~~**Decide pre-Pattern-B: should `PageMoveComplete` trigger a review-pending notification?**~~ **RESOLVED 2026-06-13** — Decision: NO. The save is the signal; the move is plumbing. Pattern A's title-drift flow is `page.move() → page.save()`, so the save always fires PageSaveComplete with the `Auto-import:` or `Auto-merge:` prefix; the move's `PageMoveComplete` would just duplicate it. The rare scenario of a content-less rename (move with no follow-up save) is captured by Special:Log (move log), the queue, and any watchlist — adding a push for it would erode the save notifications that carry actual content changes. The operator-facing run summary now logs `moved=N` directly from the bot's client-side counter. Wiki7ReviewGate keeps registering only `PageSaveComplete`; no PHP changes needed.
